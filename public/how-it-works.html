HOW IT WORKS

Video, stored as transactions

Most video lives on a server or a CDN. Here the clip itself lives on Monad. It is chopped into chunks, each written as the calldata of an ordinary transaction. To play it back, a browser reads those transactions out of the blocks and stitches them together. No storage contract, no backend.

The format

A clip becomes a sequence of packets, each carried in one transaction. There are three kinds, and every clip is exactly one metadata packet, then many stream packets, then one end packet.

MetadataDescribes the clip: mime type, file name and size, and whether it can play while it downloads. Its block is where the clip begins.
StreamA chunk of the file plus its position, the byte offset of that chunk in the whole clip. Chunks can land out of order; the position is what puts them back together.
EndThe final chunk and the total length, so the reader knows the clip is complete.

Every packet starts with the same five-byte header, so a reader can recognise a packet at a glance and ignore everything else in the block.

// 5-byte header on every packet ┌─────────────────┬──────────────┬──────────────┐ │ Magic (3 bytes) │ Version (1) │ Type (1) │ │ "dvf" │ 0x01 │ 0x00..0x02 │ └─────────────────┴──────────────┴──────────────┘ // a stream packet's calldata header(5) + position(8, little-endian uint64) + chunk bytes
Signing without the popups

A clip can be hundreds of transactions, and nobody wants hundreds of wallet prompts. So you connect your wallet once to fund a throwaway session key generated in your browser. That key signs and broadcasts every chunk itself, straight to the RPC, with no further prompts.

The session key lives in this browser's localStorage. It is a hot key, so you only fund what a clip will cost. You can also skip the wallet entirely and send funds to the session address from anywhere.

Uploading a clip
STEP 01

Prepare

WebM is ready as-is. An mp4 or mov is remuxed, or transcoded, in the browser into a form that can play while it downloads.

STEP 02

Write

The metadata packet goes out first; the block it lands in marks the start of the clip. Then the file streams out as byte-contiguous chunks, one transaction each, and an end packet closes it.

STEP 03

Share

You get a short stream code. It is just a pointer: who wrote the clip and which block to start reading from.

STEP 04

Replay

Anyone with the code reads the clip back out of the chain and watches it. Nothing was uploaded to a server at any point.

Playing it back

The reader walks the blocks from the start block forward. In each one it keeps only the transactions from the clip's sender, checks the magic bytes, and drops everything else. Chunks are reassembled by their position, so out-of-order arrival is fine.

Streamable formats play progressively — WebM, and the fragmented MP4 that mp4 and mov files are remuxed into on upload. The reader feeds chunks to the player as they arrive, so it starts after a second or two instead of waiting for the whole file. Anything that couldn't be made streamable downloads fully, then plays.

Writing bytes as calldata is not free. Monad charges the EIP-7623 calldata floor, so a clip's cost scales with its size, and on mainnet that is real MON. Keep clips short.
